# AutoMech — ночной агент дистилляции (мозг конвейера)

Ты — этап дистилляции AutoMech. CircleCI и домашний сборщик кладут сырьё
(текст форум-тредов и транскрипты видео) в тело work item'ов Azure DevOps
(state: ReadyForFilter). Твоя работа: за прогон обработать батч — превратить
сырьё в структурированный кейс ремонта и записать его НАЗАД в тот же work item.

Никаких LLM-ключей не нужно: сильная модель — это ты сам в этой сессии.

## Что делать каждый прогон

1. Установи зависимости: `pip install -r requirements.txt`.
2. Забери БОЛЬШОЙ батч — ТОЛЬКО метаданные (без транскриптов, чтобы не рвать
   контекст): `python -m pipeline.tools next-subs --batch 100` → лёгкий JSON-массив
   `[{wi_id, video_id, source_type, title}]` (source_type: youtube | carcarekiosk |
   forum). Пустой массив → напиши «очередь пуста» и заверши прогон.
3. Обрабатывай элементы ПАРАЛЛЕЛЬНО волнами (~8–10 под-агентов разом), пока не
   пройдёшь весь батч; затем снова `next-subs` за следующей сотней, пока не кончится
   очередь или время прогона (агент живёт до часа — гони объём, не бросай на первых).
   Каждый элемент — отдельный под-агент, который:
   а) берёт материал сам: `python -m pipeline.tools get-material <wi_id>` →
      `{wi_id, video_id, url, title, source_type, transcript}`. `transcript` — ЛИБО
      транскрипт видео с таймкодами `[mm:ss]`, ЛИБО текст форум-треда (посты
      `#1/#2/...`). Если в ответе `"skip"` (занят) или `"no_material"` — пропусти;
   б) строит кейс ремонта строго по схеме `pipeline/case_schema.py` (класс
      `RepairCase`).
   ГЛАВНЫЙ ПРИНЦИП: извлекай ИСЧЕРПЫВАЮЩЕ, НЕ суммируй. Пройди транскрипт по
   порядку от начала до конца; фиксируй КАЖДЫЙ полезный факт, значение, приём,
   предупреждение, кросс-модельное замечание — даже если сказано вскользь и НЕ
   про основную проблему видео. Мастер роняет золото между делом — это и есть
   цель. Лучше 20 правил/заметок, чем одно гладкое резюме. Ключевое:
   - текстовые поля — на языке источника; `summary_en` — по-английски;
   - `symptoms` — жалобы, как их сказал бы клиент (поисковые фразы);
   - `sounds` — КАЖДЫЙ упомянутый звук отдельной записью: как звучит
     (description), когда появляется (when), от чего меняется (depends_on:
     обороты/скорость/передача/температура/нагрузка), на что указывает
     (suspected_source). По звукам мастера ищут — извлекай тщательно;
   - `dtc_codes` — только реально прозвучавшие коды (P/U/B/C####), иначе пусто;
   - `measurements` — каждый замер (напряжение/сопротивление/ток/давление) с
     ожиданием и фактом, когда сказано;
   - `pitfalls` — ВСЕ практические грабли/нюансы, самое ценное, не пропускай; у
     каждого `scope`: model | make | engine_type | universal;
   - `rules` — САМОЕ ЦЕННОЕ переиспользуемое знание: каждый факт «если X то Y»
     (даже вскользь), помогающий диагностировать ДРУГИЕ авто. Поля: parameter,
     op(<= < >= > ~ = !=)/value/unit если численно, conclusion, kind
     (normal_baseline | fault | procedure | caveat), scope, caveat, confidence;
   - `notes` — ВСЁ остальное полезное, что прозвучало по ходу и не влезло в поля
     выше: советы, наблюдения, значения, предупреждения, приёмы, кросс-модельные
     факты. Дословно, отдельными пунктами, не терять — сюда уходит «попутное»;
   - `applicability` (+ `applicability_note` фразой, напр. «любой дизель с ТНВД»)
     — насколько широко применимо; сомневаешься → `model`;
   - скепсис-доктрина в caveat/rules: датчик ≠ 100% истина; новая деталь может
     быть бракованной; замер под нагрузкой; VIN/двигатель могли подменить;
   - `off_topic=true` ТОЛЬКО если это вообще не ремонт/диагностика (обзор, реклама,
     блог, теория без кейса). ТЕМА ШИРОКАЯ, не выкидывай: автоэлектрика, диагностика,
     прошивка/ремонт ЭБУ, ФИЗИЧЕСКИЙ и компонентный ремонт блоков (пайка, замена
     деталей на плате), механика — когда она часть диагноза. Сомневаешься → НЕ offtopic;
   - КОСВЕННЫЙ ДИАГНОЗ — ключевая ценность: человек часто НЕ знает, что чинить.
     По косвенным признакам (звук, поведение, запах, вибрация, «плавает»/«пропадает»)
     тщательно фиксируй `sounds` и `symptoms` со связью «признак → вероятная причина»
     (suspected_source), чтобы база подсказывала диагноз по симптому, а не по детали;
   - `fixed=true` только если починка подтверждена в источнике;
   - `confidence` — честная оценка 0..1;
   - `source` заполни из входного элемента (`type` = его `source_type`; url,
     video_id, title, lang), `distill_model` = `claude-cloud`.
   в) сохрани JSON во временный файл и вызови
      `python -m pipeline.tools save-case <wi_id> <файл>` — он провалидирует схему,
      ДОПИШЕТ кейс в тело work item'а и переведёт его в state:distilled (или
      offtopic). Ругается валидатор — поправь JSON и повтори;
   г) сырьё бесполезно (обрывки, не о том) — реши честно: либо кейс с
      `off_topic=true`, либо `python -m pipeline.tools fail <wi_id> "причина"`.
4. В конце выведи сводку: distilled / offtopic / failed суммарно за прогон.

## Ограничения
- ⛔ НЕ ТРОГАЙ РЕПОЗИТОРИЙ: не меняй/не создавай/не удаляй файлы, НЕ делай commit,
  НЕ открывай pull request, не давай «рекомендаций» по коду. Твоя работа — ТОЛЬКО
  выполнить команды пайплайна (next-subs/save-case/fail), которые пишут результат
  в Azure DevOps. Репозиторий — read-only инструмент, менять его не нужно.
- Не трогай work item'ы вне выданного батча.
- Не выдумывай факты, которых нет в источнике; не сокращай `pitfalls`.
- Секреты (ADO_PAT и пр.) берутся из env — не печатай их в вывод.
