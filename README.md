# AutoMech — база знаний автоэлектрика из YouTube и форумов

**Продукт:** мобильное приложение + API, где автоэлектрик ищет по симптому,
марке/модели и DTC-коду — и получает дистиллированные кейсы ремонта
(симптомы → замеры → первопричина → шаги → грабли) со ссылками на момент
видео-первоисточника. Кросс-языковой поиск: испанский ролик находится по
русскому запросу.

**Ключевая ставка:** ценность не в титрах, а в *структурированной выжимке*
(кейсе). Титры — сырьё, кейс — товар.

**Костяк знаний — рулбейз «если→то» (чтобы причина не терялась).** Ключевая
идея (docs/rulebase.md): общие факты из видео («на VW педаль ≤85% — норма»,
«завоздушен ТНВД → стравить: любой дизель») извлекаются как явные
**DiagnosticRule** и подставляются консультанту гарантированно, а не «если LLM
вспомнит». Это мультифлоу, не жёсткое дерево: вход {марка, параметр, значение}
активирует релевантные правила; правило другой марки приходит кросс-марочной
подсказкой. Плюс всегда — 5 правил-**скепсиса** (новая деталь ≠ рабочая; VIN мог
быть перебит — объём только замером; показание ≠ исправность; и т.д.).
`POST /consult` проверен вживую: VW+84% → «норма, не трогай»; каждый ответ
кончается явным «главное подозрение сейчас: …».

Два свойства схемы кейса, которые делают базу умнее простого поиска:
- **Область применимости** (кейс целиком + каждый нюанс): `model` → `make`
  («на всех VW датчик педали газа даёт максимум ~75%») → `engine_type»
  («завоздушен ТНВД → стравить воздух: любой дизель») → `universal`.
  Фильтр «моя машина» не отсекает универсальные знания — они подмешиваются
  в выдачу для любой марки.
- **Звуковые симптомы** отдельной сущностью: как звучит / когда / от чего
  меняется (обороты, скорость, передача, температура) / на что указывает.
  Запрос «шорох меняется при переключении передачи» находит «подшипник КПП».

## 1. Конвейер (твоя схема, уточнённая)

Всё в облаке; домашняя машина — ТОЛЬКО мониторинг (ярлык на дашборды).

```
[ADO delta-hourly, раз в час, hosted-агент]  controller delta
    поиск дельты: по активным Epic-каналам новые видео (YouTube Data API,
    DC IP ок) -> Task в New. Дедуп: Custom.url (exists_url) + дети чанков.
[ADO dispatch-hourly, раз в час :30]  controller dispatch
    смотрит очередь по состояниям -> триггерит CircleCI, РАСПРЕДЕЛЯЯ прогоны
    между аккаунтами по остатку бюджета минут (round-robin). Конфликтов нет:
    claim по System.Rev.
[ADO discovery-weekly, раз в неделю]  поиск НОВЫХ каналов по ~24 языкам -> Epic
        |
        v
[CircleCI, ПАРНЫЕ аккаунты: A чётные минуты/partition=even, B нечётные/odd]
    state:new -(claim по rev)-> транскрипт ЦЕПОЧКОЙ провайдеров:
       yt-dlp (сдохнет на DC IP - ок) -> Invidious-зеркала -> Supadata API
       (+ опция YTDLP_PROXY residential)
    архив -> R2/B2 -> state:subs
        |
        v
[Дистилляция — два равноправных воркера по одной очереди]
    ТИР 0: Claude Code cloud на ПОДПИСКЕ юзера (scheduled-агент по
      claude/DISTILL_AGENT.md + pipeline/tools.py: next-subs -> Claude сам
      пишет кейс -> save-case). Самая сильная модель, ноль внешних ключей.
    ТИР 1: CircleCI job distill — каскад API (minimax-m2.7/nemotron на NIM,
      Groq/Cerebras/Gemini, DISTILL_ENDPOINTS) для объёмов сверх подписки.
    state:subs -> RepairCase JSON (симптомы, DTC, замеры, шаги, НЮАНСЫ
    с таймкодами) -> R2/B2 -> state:distilled
        |
        v
[CircleCI, job индексации]
    state:distilled -> bge-m3 -> 1024d вектор -> Qdrant (+Oracle 23ai позже)
    эмбеддер по цепочке: Kaggle T4 сервер (если поднят) -> Cloudflare -> NIM
    -> state:indexed
        |
        v
[Oracle Free Tier ARM VM: backend/app.py (FastAPI) — НАПИСАН и проверен]
    /search: вектор (Qdrant) + фильтры make/model/system/DTC/lang,
             фолбэк — лексический поиск по cases.jsonl (работает без Qdrant)
    /answer: RAG — найденные кейсы -> БЕСПЛАТНЫЙ ЧАТ формулирует ответ
             (ANSWER_ENDPOINTS: Pollinations без ключа -> NIM -> гейтвей чатов)
    /case/{id}, /health, мини-веб на /
        |
        v
[Мобилка Flutter: mobile/ — скелет написан]
    поиск+фильтры, ИИ-ответ, карточка кейса с ▶-кнопками на таймкоды YouTube

[Форумы, фаза 7]: HAR сессии -> pipeline/forums.py (XenForo/phpBB/vBulletin)
    -> те же дистилляция и индекс, Source.type=forum
```

**ADO Boards как «база данных» (иерархия):** канал/форум = **Epic**
(`[ch:UCxxx]`/`[forum:host]`, теги `kind:channel|forum`, `state:active|paused`),
видео/тред = **child-Issue** под своим эпиком с конвейерными тегами
`state:new → subs → distilled → indexed` (`failed`, `offtopic`). Бэклог уровня
Epics с развёрнутыми детьми — витрина всей базы; rollup-колонка показывает
процент обработанности канала. Курирование = тег: мусорный канал →
`state:paused`, и синк его пропускает. Кастомизация процесса не нужна;
захочешь красоту — inherited process (см. docs/ado-setup.md, маппинг «код↔шаблон»).

**Тела данных в ADO не живут**: титры и кейсы — в объектном хранилище (R2/B2),
в work item — статус, ссылки, короткий комментарий. Иначе упрёмся в лимиты
attachment'ов и API. ADO = структура+статусы, R2/B2 = данные, Qdrant/Oracle = поиск.

## 4. Фазы

**Фаза 0 — валидация ценности (сейчас, 1-2 дня).**
Прогнать 15-20 видео на 3-4 языках: `python -m pipeline.run_phase0 --search "..." -n 5`.
Критерий: ≥70% кейсов «полезны на глаз» (симптомы/причина/нюансы верные).
Если нет — крутим промпт/модель, инфраструктуру НЕ строим.

**Фаза 1 — discovery + контроллеры (готово, ✅ протестировано на живом ADO).**
Три ADO-пайплайна:
- `delta-hourly.yml` (`controller delta`, раз в час) — дельта по активным
  каналам → Task'и в New;
- `dispatch-hourly.yml` (`controller dispatch`, раз в час :30) — по очереди
  триггерит CircleCI, распределяя прогоны между аккаунтами по остатку бюджета
  минут (round-robin, claim снимает конфликты) — **проверено вживую**;
- `discovery-weekly.yml` — поиск новых каналов на ~24 языках.
**Свои каналы**: `my_channels.txt` → `youtube_discovery --my-channels`; перепарс
— `pipeline.tools reparse-channel <id>` (дети эпика → ReadyForFilter). Курирование
мусорных каналов — тег `state:paused` на Epic.

**Фаза 2 — титры на потоке (облако).** GitHub + парные CircleCI-аккаунты
(even/odd, см. docs/circleci-setup.md), цепочка транскрипт-провайдеров:
yt-dlp → Invidious → Supadata (+YTDLP_PROXY при желании). Здесь же надо
ТЕСТИРОВАТЬ реальную проходимость Invidious-зеркал из CI и подобрать живой
набор инстансов/прокси — главный операционный эксперимент фазы.

**Фаза 3 — дистилляция.** Тир 0 — **Claude Code cloud на подписке юзера**:
scheduled-агент (промпт готов: `claude/DISTILL_AGENT.md`, инструменты —
`pipeline/tools.py`: `next-subs`/`save-case`/`fail`); юзер настраивает routine
сам. Тир 1 — CircleCI job `distill` с каскадом `DISTILL_ENDPOINTS`
(NIM/Groq/Cerebras/Gemini/Pollinations) на объёмы сверх подписки. Оба воркера
живут на одной очереди ADO без конфликтов (claim + партиции).

**Фаза 4 — индексация.** job `embed-index`: bge-m3 → Qdrant. Эмбеддер по
цепочке: Kaggle T4 сервер (kaggle/embedding_server.py, поднимать на большие
батчи) → Cloudflare → NIM. Позже Oracle 23ai как вторая поисковая реплика.

**Фаза 5 — API. ✅ НАПИСАН и проверен вживую** (`backend/`): `/search`
(Qdrant-вектор + фильтры; лексический фолбэк по cases.jsonl — работает до
настройки Qdrant), `/case/{id}`, `/answer` (RAG: кейсы + бесплатный чат из
каскада ANSWER_ENDPOINTS — идея юзера «чаты формируют ответ на бекэнде»;
Pollinations без ключа проверен), мини-веб на `/`, Dockerfile под Oracle ARM.
Проверено на кейсах Фазы 0: поиск находит ХОВО-кейс, ИИ-ответ цитирует его
с реальной первопричиной. Осталось: развернуть на Oracle VM.

**Фаза 6 — мобилка. Скелет написан** (`mobile/`): Flutter, два экрана —
поиск (фильтры + «Спросить ИИ») и карточка кейса с ▶-кнопками, открывающими
YouTube на таймкоде шага/нюанса. Не собирался (нет Flutter SDK на этой
машине) — первая сборка по mobile/README.md. Дальше: избранное, офлайн-кэш.

**Фаза 6.5 — CarCareKiosk.com (источник №2, код готов).**
`pipeline/carcarekiosk.py`: sitemap → 1517 машин (EN; + зеркала es./fr.) ×
~80 задач ≈ **100k+ коротких видео-инструкций** с прямыми MP4 (CloudFront) и
краткими шагами в meta description. Титров нет → ASR (`pipeline/asr.py`,
цепочка: Kaggle T4 `/transcribe` whisper-large-v3 → Cloudflare
`@cf/openai/whisper` → локальный faster-whisper). robots.txt разрешает.
Иерархия: Epic на марку `[site:cck-Make]` → child-задачи. Сайт переиспользует
видео между машинами-близнецами (Acura CL → ролик Honda Accord) — дедуп по
mp4-URL при желании. Экономика ASR: ролики 1-4 мин, large-v3 на T4 ≈
15-20× реалтайм → ~2-3 тыс. роликов за 30 GPU-ч/нед.

**Фаза 7 — форумы. Краулер написан и проверен** (`pipeline/crawler.py` +
`forum_sites.py`, docs/crawler.md). Обходит все треды форума, посты → R2,
каждый тред → тикет ADO (child под Epic форума, сразу `state:subs`) → дальше
дистилляция берёт наравне с видео.
- **Зоны** (`forum_sites.py`): сайты разбиты на зоны a/b/c; прогон обходит одну
  зону → разные CI-аккаунты бьют по разным хостам в разное время, IP не банится.
- **Вежливость**: пауза 6 c на хост + чередование хостов (нагрузка размазана).
- **Тайм-бокс** `--minutes 18`: прогон выходит по дедлайну, сохранив фронтир в
  R2; следующий продолжает. Укладывается в бюджет CI-джоба (<20 мин).
- Движки: IPS/XenForo/vBulletin/phpBB + кастомные drive2/opinautos (detect
  чинён — скан всего HTML). Живой тест зоны A: раздел BMW carmasters → 31 тред
  + пагинация, вытащены реальные кейсы («коробка в аварию, заклинил термостат»,
  «ошибки кодирования ACSM4 F30»), 229 URL в очереди, всё возобновляемо.
- Только текст постов, без копий медиа (контент-политика §5.5); за логин — `--har`.

Источники и вердикт по всем 11 HAR-дампам из `dump/` — в **docs/sources-scan.md**.
motor-talk (GraphQL) и reddit (OAuth 403 без ключа) — отдельные клиенты, не краул.

**Фаза 7.5 — осциллограммы.** Свободные библиотеки осциллограмм (Open Labor
Project — бесплатно, 104 машины с «эталон/сигнатура отказа»; PicoScope —
тысячи community) ложатся прямо в рулбейз: эталонный паттерн = `normal_baseline`,
сигнатура отказа = `fault`. `pipeline/waveforms.py` — конвертер (без
дистилляции). Берём факты-диапазоны, не копии картинок (контент-политика §5.5).

**Фаза 8 — запчасти и СТО (монетизация).** Кейсы уже извлекают `parts`
(с OEM-номерами, когда звучат). Поверх: скан рынков (EU/US/UA…) на наличие
и цены этих запчастей (каталоги типа autodoc, RockAuto; партнёрские ссылки) +
каталог СТО по специализациям с гео («шорох при переключении → подшипник КПП →
ближайшие СТО, которые делают именно коробки», источники: Google Places/2GIS).
Это и есть монетизация: лиды на запчасти и сервисы к моменту, когда мастер уже
знает, ЧТО менять.

## 5. Структура репо

```
pipeline/
  config.py                все настройки из .env
  case_schema.py           СХЕМА КЕЙСА (главный артефакт продукта)
  subtitles.py             yt-dlp + нормализация VTT (проверено на реальных титрах)
  subtitle_providers.py    транскрипт для облачного CI: ytdlp -> Invidious -> Supadata
  distill.py               титры -> RepairCase (любой OpenAI-совместимый endpoint)
  embed.py                 bge-m3: Kaggle-сервер -> NIM -> Cloudflare -> локально
  ado.py                   work items: WIQL, state:* тегами, claim, партиции even/odd
  youtube_discovery.py     seed-поиск на 24 языках + дельта; реестр в R2
  forums.py                форумы: HAR-сессия, XenForo/phpBB/vB/IPB, та же дистилляция
  store.py                 JSONL + R2/B2 + Qdrant (upsert идемпотентен по video_id)
  run_phase0.py            сквозной прогон для валидации
```

## 5.5 Контент-политика (что показываем, что нет)

Принцип: **мы справочник кейсов со ссылками на первоисточники, а не хостинг
чужого контента**. Факты (симптомы, замеры, причины, шаги) не охраняются
авторским правом — кейс-выжимка наша. Выражение (видео, кадры, картинки,
дословный текст) — охраняется, его не копируем и не перезаливаем.

- YouTube: официальный **embed-плеер** с `?start=<сек>` в вебе, deep link в
  приложение YouTube в мобилке. Разрешено ToS, канал получает просмотры.
- CarCareKiosk: только ссылка на их страницу задачи. Прямой mp4 в UI не
  светим, своим плеером не оборачиваем (обход их страницы/рекламы).
- Картинки/кадры: не копируем и не хотлинкаем; иллюстрации — свои пиктограммы
  систем; превью — стандартный link-preview паттерн, всегда ведёт на источник.
- Транскрипты (титры, ASR) — внутреннее сырьё дистилляции, наружу не отдаём.
- Takedown с первого дня: по запросу правообладателя кейс скрываем.
- Целевое состояние — официальные партнёрства с каналами/CCK, когда появятся
  метрики трафика («мы привели вам N переходов») — им это выгодно.

## 6. Риски и честные оговорки

1. **ToS YouTube.** Массовое выкачивание титров нарушает ToS YouTube. Риск
   прагматично снижаем: качаем только титры (не видео), всегда линкуем
   первоисточник с таймкодом (каналам это трафик), храним выжимку, а не
   контент. Юридическую позицию (цитирование/fair use в целевых юрисдикциях)
   проработать до публичного запуска. План Б: официальные интеграции с
   каналами (рев-шара за трафик) — им это выгодно.
2. **429/блокировки.** Главный операционный риск при cloud-only схеме.
   Митигация — цепочка транскрипт-провайдеров (yt-dlp → Invidious-зеркала →
   Supadata) + 1 дорожка/видео + паузы; радикально — YTDLP_PROXY
   (residential, ~$1-3/GB). Invidious-инстансы смертны — список в env,
   реальную проходимость из CI обязательно тестировать в фазе 2.
3. **Качество авто-титров.** Спецтермины распознаются криво («стэн» вместо
   «стенд»). Частично лечится промптом (модель восстанавливает по контексту),
   позже — Whisper large-v3 по аудио для топ-каналов (есть на NIM бесплатно).
4. **Лимиты бесплатных LLM.** Решается каскадом провайдеров (клоны твоего
   `api_providers.py`) и ночными батчами. Всегда сохраняем сырьё — можно
   пере-дистиллировать лучшей моделью позже.
5. **ADO как БД.** До ~50-100k work items — ок. Дальше Boards станет тесно:
   миграция статусов в Postgres/Oracle тривиальна (states уже в JSON-архиве).
6. **Секреты.** В `ClaudeCode/nvidea/keys.py` и `Free/*.txt` живые ключи
   плейнтекстом. Сюда — только `.env` (в .gitignore), в CI — variable
   groups/contexts. Старые засвеченные ключи ротировать.
