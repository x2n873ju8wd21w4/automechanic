# Костяк знаний: рулбейз «если → то» (чтобы причина не терялась)

Проблема: во время видео/треда всплывают ОБЩИЕ факты («датчик педали на VW
больше 85% не даёт», «завоздушен ТНВД → стравить»). Если полагаться только на
векторный поиск и память LLM — такой факт теряется, и консультант уводит
мастера не туда. Решение — **рулбейз**: явные структурированные правила,
которые подставляются в ответ гарантированно.

Это НЕ жёсткое дерево-флоучарт (мир слишком разнообразен), а **мультифлоу**:
набор правил-узлов, вход активирует релевантные.

## Правило (`DiagnosticRule` в case_schema.py)

```
parameter   что наблюдаем: "датчик педали газа, %" | "давление в рампе" | звук | DTC
condition   человекочит.: "максимум ~85%"      op/value/unit  структурно: <= 85 %
conclusion  что значит: "норма, фишка VAG"
kind        normal_baseline | fault | procedure | caveat
scope       model | make | engine_type | universal        make/model  привязка
confidence  0..1                                caveat  скепсис к этому факту
```

Правила извлекает дистилляция из КАЖДОГО кейса (промпт в distill.py), плюс
«осциллограммы» (waveforms.py) вливаются правилами напрямую.

## Как активируется (`rulebase.match`)

Вход: `{make, parameter, value, unit, query_text}`. Для каждого правила:
- **применимость к машине** → relation:
  - `direct` — правило той же марки (или узкое, совпало);
  - `general` — universal / engine_type (применимо всем);
  - `hint` — правило ДРУГОЙ марки → подсказка «на VW так; у тебя другая марка,
    возможно похоже — уточни».
- **числовой вердикт** — если у правила есть `op/value` и клиент дал значение:
  проверяем условие. Пример: правило VW `<=85%`, клиент 84% → «84% ≤ 85% →
  норма».
- caveat-правила скепсиса (`doctrine/skepticism_rules.json`) добавляются ВСЕГДА.

Проверено: VW + 84% → `direct` «это норма»; Toyota + 84% → `hint` «на VW норма,
у тебя другая марка — уточни». Ровно поведение из ТЗ.

## Скепсис (doctrine/skepticism_rules.json + claude/CONSULT_DOCTRINE.md)

Пять универсальных caveat'ов, всегда в контексте консультанта:
1. Новая деталь ≠ рабочая (заводской брак) — не вычёркивать узел.
2. Показание датчика ≠ доказательство исправности — мерить под нагрузкой.
3. VIN/двигатель могли перебить — объём подтверждать замером поршня, не VIN.
4. «Проверено/заменено» — гипотеза, а не факт — держать в подозреваемых.
5. Нет оборудования — давать метод «на коленке» рядом с инструментальным.

Доктрина — редактируемый JSON + markdown; расширяй свободно.

## Консультация (`POST /consult`)

```
{ "make":"Volkswagen","model":"Passat","engine":"2.0 TDI",
  "question":"педаль газа максимум 84%, это норма?",
  "observations":[{"parameter":"датчик педали газа","value":84,"unit":"%"}],
  "dtc":["P2135"] }
```
Бекэнд: (1) векторный поиск кейсов по машине (с учётом применимости);
(2) `rulebase.match` по каждому замеру/коду/жалобе → direct/hints/general/caveats;
(3) LLM (ANSWER_ENDPOINTS) с доктриной CONSULT_DOCTRINE.md собирает ответ,
подставляя правила ЯВНО. В ответе — `rules_used` (что сработало) и ссылки на кейсы.

Так причина держится костяком, а LLM только формулирует — не «вспоминает».
